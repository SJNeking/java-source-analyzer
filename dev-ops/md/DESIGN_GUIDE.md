# Java 项目代码智能可视化工作台 - 设计指南

## 一、核心设计理念

### 1.1 产品定位

这不是一个简单的"画图工具"，而是一个**代码智能工作台（Code Intelligence Workbench）**，需要同时解决四类问题：

- **看全局**: 项目有哪些组件、组件之间怎么依赖、哪些模块复杂脆弱耦合高
- **看局部**: 某个类里有哪些字段方法、方法之间怎么互调、这个类为什么难维护
- **看链路**: 一个方法调了哪些方法、调用链路如何跨组件传播、哪些链路风险高
- **看质量**: 哪些类/方法有问题、问题严重性如何、问题如何映射到架构链路组件

### 1.2 五个可视化层级

```
L1. 项目层    → 模块、包、组件数量、总体质量、热点区域、入口点
L2. 组件层    → Java类/接口/抽象类/枚举、字段、方法、依赖、质量画像
L3. 字段层    → 字段名、类型、访问修饰符、注解、用途标签、被多少方法使用
L4. 方法层    → 方法签名、参数、返回值、标签、行数、复杂度、调用关系、质量问题
L5. 调用链层  → 方法到方法、组件到组件、入口到出口、跨层调用、风险传播路径
```

## 三、六个核心可视化视图

### 视图1: 全局组件依赖图（力导向图）

**目标**: 看整个Java项目由哪些组件组成，组件之间如何依赖。

**节点编码**:
- **颜色**: 组件类型（Controller🌐 / Service⚙️ / Repository💾 / Entity🏛️）
- **大小**: 复杂度 / 被依赖度 / 方法数
- **边框颜色**: 质量风险等级（红=CRITICAL / 橙=MAJOR / 蓝=MINOR / 绿=无问题）
- **光晕**: 热点组件
- **图标**: 组件类型标识

**边类型**:
- `DEPENDS_ON` - 依赖
- `IMPLEMENTS` - 实现
- `EXTENDS` - 继承
- `CALLS` - 调用
- `INJECTS` - 注入
- `USES` - 使用

**交互**:
- 缩放/平移
- 搜索节点
- 过滤类型
- 按包折叠
- 点击打开详情
- 双击进入组件内部视图

**技术选型**: ECharts Graph（中小规模）或 Cytoscape.js/AntV G6（复杂图交互）

---

### 视图2: 组件内部结构图（UML风格卡片）

**目标**: 看"一个类内部"的字段、方法和内部关系。

**方案A: UML风格卡片图**

```
┌─────────────────────────────────┐
│  🔷 UserService                 │
│  @Service                       │
├─────────────────────────────────┤
│  - userRepository: UserRepository │
│  - transactionManager           │
├─────────────────────────────────┤
│  + createUser(user: User): void │
│  + updateUser(user: User): void │
│  - validateUser(user: User): bool│
└─────────────────────────────────┘
```

**方案B: 组件内部关系图**

- **节点分三类**: 类（中心）、字段（左）、方法（右）
- **边类型**:
  - 类 → 字段（has field）
  - 类 → 方法（has method）
  - 方法 → 字段（reads/writes，区分颜色）
  - 方法 → 方法（calls internal method）

**视觉编码**:
- 类节点居中
- 字段在左，方法在右
- 内部调用线用细边
- 字段读写区分颜色（读=蓝色，写=红色）

---

### 视图3: 方法调用图

**目标**: 看方法到方法的调用关系。

**两种模式**:

**模式A: 单类内部方法图**
- 专门看一个类的方法互调
- 节点 = 方法
- 边 = CALLS（带权重）

**模式B: 跨组件方法调用图**
- 专门看跨类调用
- 节点可按所属类分组
- 边显示调用方向和频次

**关键策略**:
- **深度限制**: 默认只展开 N 层
- **按需展开**: 点击节点加载下游
- **焦点模式**: 只看某个方法的上游/下游
- **高亮递归/循环调用**

**警告**: 方法图最容易爆炸，默认不要全展开！

---

### 视图4: 调用链路追踪图（分层泳道）

**目标**: 回答"一个入口方法到底调到了哪里，中间经过哪些组件，哪些环节有质量问题"。

**方案A: Sankey流向图**
- 适合看从入口到下游的调用扩散
- 节点宽度反映调用频次

**方案B: 时序/泳道图**
- 适合看 Controller → Service → Repository → External API
- 每层一列，调用线从左往右走

**方案C: 分层调用链图**
```
Controller层    Service层      Repository层    DB/External
    │              │                │               │
    ├─────────────→│                │               │
    │              ├───────────────→│               │
    │              │                ├──────────────→│
```

**质量叠加**（强烈推荐）:
- 🔴 红点 = 高危方法
- 🟡 黄色边 = 跨层违规
- ⚪ 虚线 = 未解析调用
- 🔒 锁图标 = 安全问题
- 🔥 火焰图标 = 性能热点

---

### 视图5: 架构分层矩阵

**目标**: 看架构是否越层、模块边界是否合理。

**方案A: 层间依赖矩阵**

```
           Ctrl  Svc  Repo  Entity  Util
Controller   -    ✅    ❌     ❌     ✅
Service      ✅    -    ✅     ✅     ✅
Repository   ✅   ✅     -     ✅     ✅
Entity       ❌   ✅    ✅     -     ❌
Util         ✅   ✅    ✅     ❌     -
```

单元格显示调用/依赖次数，违规则标红。

**方案B: 层级节点图**
- 按层垂直/水平排布
- 上层到下层的关系可视化
- 违则红色边显示

**质量检测点**:
- Controller → Repository 直接调用标红
- Util 被过度依赖高亮
- 某层类数过大告警

---

### 视图6: 质量分析总览面板

**目标**: 把质量问题从"列表"升级成"工程风险地图"。

**不应该只做 issue list，而应该做**:

#### A. 总体质量概览
- 总问题数
- 严重级别分布（饼图）
- 类别分布（柱状图）
- 技术债估算
- 质量门禁是否通过

#### B. 风险组件榜单
- 问题最多类 Top 10
- 复杂度最高类 Top 10
- 耦合最高类 Top 10
- 最危险调用链 Top 10
- 最脆弱模块

#### C. 图谱叠加（关键）

**质量问题不能只在单独tab里，而要叠加到所有图上**:

- 节点边框颜色 = 最严重问题级别
- 节点角标 = issue count
- 边颜色 = 架构违规/高风险调用
- 面板里直接显示问题详情和修复建议

## 四、数据模型设计

### 4.1 基础实体

```typescript
interface Asset {
  id: string;
  address: string; // FQN (Fully Qualified Name)
  type: 'CLASS' | 'INTERFACE' | 'ENUM' | 'ABSTRACT';
  module?: string;
  package: string;
  annotations: string[];
  modifiers: string[];
}

interface Field {
  name: string;
  type: string;
  modifiers: string[];
  annotations: string[];
  usedByMethods: string[]; // 方法地址列表
}

interface Method {
  name: string;
  signature: string;
  parameters: Array<{name: string; type: string}>;
  returnType: string;
  modifiers: string[];
  annotations: string[];
  linesOfCode: number;
  cyclomaticComplexity: number;
  callsInternal: string[]; // 内部调用方法地址
  callsExternal: string[]; // 外部调用方法地址
  calledBy: string[]; // 被谁调用
}

interface Relation {
  source: string; // 源组件/方法地址
  target: string; // 目标组件/方法地址
  type: 'DEPENDS_ON' | 'IMPLEMENTS' | 'EXTENDS' | 'CALLS' | 'INJECTS';
  weight?: number; // 调用次数/依赖强度
}

interface QualityIssue {
  id: string;
  severity: 'CRITICAL' | 'MAJOR' | 'MINOR';
  category: 'COMPLEXITY' | 'DUPLICATION' | 'SECURITY' | 'PERFORMANCE' | 'MAINTAINABILITY';
  message: string;
  location: {
    asset: string;
    method?: string;
    line?: number;
  };
  recommendation?: string;
}

interface Metrics {
  loc: number; // Lines of Code
  cyclomaticComplexity: number;
  fanIn: number; // 被调用数
  fanOut: number; // 调用他人数
  coupling: number; // 耦合度
  cohesion: number; // 内聚度
}
```

### 4.2 前端统一数据模型

```typescript
interface ProjectData {
  metadata: {
    name: string;
    version: string;
    analyzedAt: string;
  };
  assets: Asset[];
  fields: Record<string, Field[]>; // key = 类地址
  methods: Record<string, Method[]>; // key = 类地址
  relations: Relation[];
  qualityIssues: QualityIssue[];
  metrics: Record<string, Metrics>; // key = 资产地址
}
```

## 五、交互设计原则

### 5.1 定位交互（3秒找到对象）

必须支持：
- 全局搜索组件
- 搜索方法
- 搜索字段
- 搜索 issue
- 搜索包/模块

搜索结果要支持：
- 直接跳转
- 高亮在图中定位
- 在树中展开
- 在右侧显示详情

### 5.2 聚焦交互（避免图谱爆炸）

所有图都应该支持：
- 仅看上游
- 仅看下游
- 仅看 1~N 跳
- 仅看当前包
- 仅看当前模块
- 仅看有问题节点
- 仅看高复杂度对象

**这是避免图谱爆炸的关键！**

### 5.3 下钻交互（固定路径）

推荐固定一条下钻路径：
```
项目 → 模块 → 包 → 组件 → 方法 → 调用链
```

例如：
- 点击模块 → 进入模块内组件图
- 点击组件 → 进入组件结构视图
- 点击方法 → 进入方法调用图
- 点击调用边 → 进入调用链追踪

### 5.4 解释交互（告诉用户"为什么"）

不能只高亮，要解释"为什么"。

例如点击红色组件时，右侧直接显示：
- 为什么它是红色
- 有几个 critical issue
- 哪些方法最危险
- 为什么复杂度高
- 它依赖了哪些高风险组件

### 5.5 比较交互（后续扩展）

很有价值，建议后续支持：
- 两个组件对比
- 两个版本对比
- 本次扫描 vs 上次扫描
- 问题变化趋势

## 七、如何把质量分析融入可视化

### 7.1 在组件节点上叠加质量

**视觉建议**:
- 红色边框 = 有 critical
- 橙色边框 = 有 major
- 节点右上角 badge = 问题总数
- 节点透明度 = 可选表示健康度
- 节点大小 = 可叠加问题密度

### 7.2 在方法节点上叠加质量

**视觉建议**:
- 方法项前加 severity 小圆点
- 高复杂度方法用火焰图标 🔥
- 安全风险方法用盾牌/警告图标 🛡️
- 资源泄露/异常处理用专用图标

### 7.3 在调用链上叠加质量

**视觉建议**:
- 经过高风险方法的链路高亮
- 跨层违规的边用红色虚线
- 反复循环调用用紫色回环边
- 经过敏感方法的链路加锁图标 🔒

### 7.4 在详情面板中聚合质量

组件详情里显示：
- 问题总数
- 按严重级别统计
- 按类别统计
- 影响的方法
- 推荐修复顺序

## 八、性能优化策略

### 8.1 分级加载

**Level 1**: 模块级（只看模块节点）  
**Level 2**: 包级（展开模块后看包）  
**Level 3**: 组件级（展开包后看类）  
**Level 4**: 方法级（选中组件后再显示方法）

**不要默认全展开方法级图！**

### 8.2 图谱聚合

当节点太多时：
- 把包内低价值类聚合成一个 cluster
- 把工具类折叠
- 把外部依赖节点折叠成 "External"

### 8.3 边裁剪

调用边最容易爆炸，建议支持：
- 仅显示 Top N 强依赖
- 仅显示跨组件边
- 仅显示跨层边
- 仅显示有质量问题的边

### 8.4 虚拟化与懒加载

尤其左侧树和方法列表：
- 树节点虚拟滚动
- 方法列表按需展开
- 详情面板按 tab 懒渲染

## 九、技术选型建议

### 9.1 前端框架

- **如果要认真做产品**: React + TypeScript 或 Vue 3 + TypeScript
- **如果只是快速迭代静态工作台**: 可以继续原生 TS，但长远不建议

### 9.2 图谱库

**首推**:
- **Cytoscape.js**: 关系图和调用图很适合，交互能力强
- **AntV G6**: 企业级图编辑/布局能力强

**保守路线**:
- **ECharts Graph**: 已有基础、成本低

**建议**:
- 如果你现在项目已经用了 ECharts，可以第一阶段继续 ECharts
- 第二阶段把复杂关系图迁到 G6 或 Cytoscape
- 统计图、热力图、Sankey 保留用 ECharts

### 9.3 代码查看器

- **Monaco Editor**（强，VS Code同款）
- **CodeMirror**（轻）
- **简单版**: `<pre>` + `<code>` + syntax highlight

### 9.4 UI 组件

任意成熟组件库即可（Ant Design / Element Plus），重点是：
- Tree（左侧导航）
- Table（质量问题列表）
- Drawer / Splitter（详情面板）
- Tabs（视图切换）
- Tooltip（悬停提示）

## 十二、核心设计原则总结

### 12.1 不是画一个图，而是做一个工作台

- 多层级、多视角的分析工作台
- 质量分析不是孤立的列表，而是叠加到所有视图中的风险信息

### 12.2 用户能否 3 秒定位对象，10 秒理解关系，30 秒发现问题

- 定位交互：全局搜索、快捷入口
- 聚焦交互：过滤、裁剪、聚合
- 下钻交互：固定路径、逐级深入
- 解释交互：不仅高亮，还要说明"为什么"

### 12.3 性能是成败关键

- 分级加载：不要默认全展开
- 图谱聚合：折叠低价值节点
- 边裁剪：只显示关键边
- 虚拟化：大数据量时的必要手段

### 12.4 质量必须叠加到结构中

- 节点边框颜色 = 问题严重性
- 节点角标 = issue count
- 边颜色 = 架构违规
- 详情面板 = 问题详情 + 修复建议

## 十三、最终页面布局方案

### 布局方案A：分析工作台（推荐）

```
顶部栏
├─ 项目选择器
├─ 搜索框
├─ 视图切换 Tab
├─ 全局过滤器
└─ 导出按钮

左侧
├─ 项目树
├─ 质量快捷入口
└─ 入口点/API 快捷入口

中间
└─ 当前视图画布（根据 Tab 切换）

右侧
└─ 详情/问题/指标/源码摘要
```

这是最均衡的方案。

### 布局方案B：双层模式

**模式1：全局探索**
- 重点看组件关系和质量热力图

**模式2：局部剖析**
- 重点看某个组件内部字段/方法/调用链

这个模式切换也很好，因为全局和局部的信息密度差很多。

## 附录：常见陷阱与最佳实践

### 陷阱1：图很多但不可用

**原因**: 没有做好聚焦、过滤、下钻机制  
**解决**: 强制实现深度限制、按需展开、焦点模式

### 陷阱2：质量和结构分离

**原因**: 质量只在单独 tab 显示，不和图谱联动  
**解决**: 质量信息必须叠加到所有视图中

### 陷阱3：默认全量加载

**原因**: 一次性加载所有组件、字段、方法、调用关系  
**解决**: 分级加载、懒加载、虚拟化

### 陷阱4：硬编码特定架构

**原因**: 只适配 DDD 或 MVC，无法通用  
**解决**: 自动检测架构类型，提供通用布局

### 最佳实践1：统一的 detail panel

点击任何对象（组件、字段、方法、边）都显示在右侧详情面板，保持界面一致性。

### 最佳实践2：视图切换而非页面跳转

用 Tab 切换不同视图，保持上下文连贯，避免用户迷失。

### 最佳实践3：搜索即定位

搜索结果不仅要列出对象，还要在图中高亮、在树中展开、在详情中显示。
